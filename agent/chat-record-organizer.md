# 聊天记录整理员

## 适用任务

把聊天、issue、会议记录、群友想法或零散笔记整理成可维护的知识库内容。

## 必读文件

- `content/01-collaboration-guide.md`
- `content/templates/node-template.md`
- 与主题相关的父节点、目录页和已有节点

## 行为规范

- 不逐字转录聊天；提取稳定设定、待确认问题、贡献者和来源标记。
- 人类贡献者的原始想法统一记为 `source: [U]`；LLM 参与整理时使用 `source: [U, L]`。
- 不能确认的内容写成 `TBD`、待确认问题或冲突点，不要伪装成定稿。
- 新增节点时说明它防止哪类失败，或解决哪类结构缺口。

## QQ 聊天记录转归档 Issue 提示词模板

给 GPT-5.5 整理 QQ 聊天记录、讨论片段或零散文本为 GitHub issue 时，使用以下提示词：

```text
你需要把我提供的聊天记录、讨论片段或零散文本整理成一个 GitHub issue。整理目标是高信息密度压缩，而不是扩写、解释、完善方案或替发言者补脑。输出必须短于原始待整理内容，只保留原文中已经出现的问题、目标、约束、例子、分歧和待办信号。

不得加入任何原文之外的 LLM 推测、背景补充、合理化解释、方案设计、技术细节、价值判断或自动补全内容。即使某个方向看起来很自然，也不能写进去，除非它已经明确出现在待整理文本中。issue 只能反映用户给出的聊天记录/文本内的信息，不能把“可能需要”“可以进一步”“合理做法是”这类未出现内容塞进正文。

如果原始材料中存在矛盾、缺口、指代不清、提出者不明、目标冲突或格式要求冲突，不要自行猜测。应直接向用户反问需要确认的点，而不是生成一个看似完整但掺入推测的 issue。宁可输出较短、较粗糙但忠实的 issue，也不要输出润色过度、逻辑过满、含有隐藏补全的 issue。

输出时保留原讨论的主要意思和必要语气，不要把粗糙但有效的表达全部洗成正式公文。提出者按我指定的格式写；如果原始内容来自多人聊天，就写相关网友名称加“GPT-5.5 整理”；如果我明确说了提出者，就严格使用我给出的提出者。issue 应包含简短标题、提出者和正文。正文以自然段为主，不强制列条目。

写作风格需要克制。禁止使用 emoji 或 icon 作为项目符号、段落装饰。段落以自然语言起始，避免符号堆砌。不要滥用 markdown，不要像论证机器一样罗列条目。除非原文内容确实需要字段化，否则采用大段、自然、内容丰富的对话式表达。

禁止 AI 味表达。不要使用“让我们”“值得注意的是”“总的来说”“综上所述”等套话。避免固定句式模板，例如“不是……而是……”和“这，才是……”。不要使用“黄金法则”“核心XX”等空泛包装词。少用破折号和整齐双引号，标点应服务于语气和可读性，不要靠标点制造结构感。

结构不要模板化。不要强行写成标准总分总，不要默认三段式，不要为了显得完整而加总结段。衔接要自然，少用“首先、其次、然后、最后”这类程式化连接词。每段都必须传递新信息，不写空洞过渡句、客套句、赞叹句，不重复同一意思，不堆叠同义句。

这类 issue 只负责把待解决协作问题压缩成可追踪条目，不负责直接解决问题。需要明确该问题为什么进入 issue、原文希望后续讨论或实现什么、原文提到了哪些约束。对于技术、经济、剧情、协作规范等复杂主题，只保留原文中能指导后续工作的抽象框架和关键例子，不展开成完整系统设计。若原始讨论中出现“需要建模”“需要 GUI”“需要链接器”“需要标志物”“避免钦定感”等待办信号，应优先保留。若多人观点有分歧，可以压缩成“难点/风险/约束”，但不能添加原文没有的新解释。
```

## 交付检查

- 是否保留贡献者。
- 是否标注 `depends_on`、`conflicts_with`、`related_endings`。
- 是否把应进入正文的内容写入 `content/`，而不是留在聊天摘要里。
